## Rules

### 1. Access & Membership
- Membership is granted by the senior developers or owner only
- Do not share repository access, tokens, or credentials with anyone outside the organization
- Members who are no longer active on the project will have access revoked

### 2. Branching & Commits
- `live` is protected — no direct pushes under any circumstances
- All changes go through a pull request and must be reviewed before merging
- Branch naming: `feature/`, `fix/`, `patch/`, `docs/`, `new/`, `dev/` prefixes required
- Commit messages must be descriptive — no `fix`, `update`, or `misc` commits

### 3. Pull Requests
- Every PR requires at least one approval before merging
- PRs must include a clear description of what changed and why
- Link relevant issues where applicable
- Do not merge your own PR without a second pair of eyes unless explicitly authorized

### 4. Issues & Project Tracking
- All tasks, bugs, and feature requests must be tracked as issues
- Use labels: `bug`, `feature`, `docs`, `blocked`, `in-progress`
- Do not close issues without a corresponding commit or documented reason

### 5. Code Standards
- No hardcoded credentials, API keys, or server IPs in any commit
- Use `.env.example` files — never commit `.env` directly
- Comment your code where intent is not immediately obvious

### 6. Documentation
- All significant systems must have accompanying documentation
- Keep READMEs up to date when functionality changes
- Internal design documents live in the `/docs` repository

### 7. Communication
- Development discussion stays in the designated dev channel on Discord
- Do not discuss unreleased features or internal systems in public server channels
- Tag the senior developers for anything requiring immediate attention

### 8. Conduct
- This is a professional workspace — conduct yourself accordingly
- Disputes over code or design decisions are resolved through discussion, not unilateral changes
- Repeated disregard for these rules results in removal from the organization
